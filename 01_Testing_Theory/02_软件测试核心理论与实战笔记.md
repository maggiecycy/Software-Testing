# 软件测试核心理论与实战笔记

## 一、 软件研发与测试生命周期
* **标准流程**：需求分析 -> 概要/详细设计 -> 开发编码 -> 软件测试 -> 验收/发布
* **【💡 实战补充】**：测试应**尽早介入**。在“需求分析”阶段，测试需参与需求评审，提早发现逻辑漏洞（静态测试），降低后期修复成本。

---

## 二、 质量模型与测试分类

### 1. 软件质量模型
* **功能性**：功能实现。
* **非功能性**：
  * **性能**：多用户同时访问（时间/资源）。
  * **兼容性**：不同设备/平台（Web 端常见如 Chrome、Firefox、Safari、Edge）。
  * **易用性**：用户体验交互。
  * **安全性**：数据传输与存储安全。
  * **可靠性**：持续稳定运行。
  * **可移植性**：数据属性迁移。
  * **可维护性**：出错易修复。

### 2. 测试分类
* **按生产阶段划分**：
  * **单元测试**（开发负责/代码级） -> **集成测试**（测试负责/组装级） -> **系统测试**（测试负责/全面覆盖） -> **验收测试**（用户负责/交付级）。
* **按代码可见度划分**：
  * **黑盒测试**：关注 UI/系统功能，不看代码。
  * **灰盒测试**：关注部分源码/接口级/组件组装。
  * **白盒测试**：关注全部源码/单元逻辑。
* **按执行时机划分**：
  * **冒烟测试**：针对核心功能的主干路径测试（提测后首要任务）。
  * **回归测试**：修复 Bug 或系统更新后，验证旧功能是否受影响。

## 三、 测试用例设计方法论

### 1. 单功能测试（无附加）
* **等价类划分法**（多用于表单页面元素测试）：
  * **有效等价类**：满足需求的数据。
  * **无效等价类**：不满足需求的数据。
  * **测试点提取原则**：
    * **P0（高优）**：多个有效数据组合（正向验证）。
    * **P1（中优）**：单个无效数据 + 多个有效数据组合（逆向验证/控制变量法）。

### 2. 长度与范围测试
* **边界值分析法**（多用于边界限制场景）：
  * **上点**：边界上的点。
  * **离点**：离上点最近的点。**【💡 重点纠错】如果边界是闭区间 `[a, b]`，离点在范围外；如果是开区间 `(a, b)`，离点在范围内。**
  * **内点**：任意范围内的点（中间值）。
  * **设计 Step**：确定边界值 -> 划分等价类（类型和规则） -> 提取数据组合。

### 3. 多条件约束测试
* **判定表法**（多条件组合规则）：
  * **结构**：条件桩、动作桩、条件项、动作项。
  * **推导**：所有条件 -> 可能动作 -> 判定真假值 -> 最终操作动作。
  * **规律**：有 n 个独立条件，取值 (0, 1)，则有 2^n 条规则组合。

### 4. 业务链路测试
* **流程图法**（关注模块间关联性）：
  * **执行顺序**：项目提测 -> 对主路径进行**正向冒烟测试** -> 测完整业务流程 -> 细化测试单模块。
  * **原则**：流程图中的每条路径均可转化为一个测试用例。

## 四、 测试用例文档与缺陷 (Bug) 管理

### 1. 测试用例八大要素
1. **用例编号**：项目_模块_数字。
2. **用例标题**：预期结果/测试点。
3. **所属模块**：归属的业务功能。
4. **优先级**：用例的重要程度（P0/P1/P2等）。
5. **前置条件**：执行该步骤前必须满足的环境或状态。
6. **测试步骤**：执行的具体操作。
7. **测试数据**：输入的值。
8. **预期结果**：显性执行结果 + 隐性数据落表。

### 2. 缺陷 (Bug) 生命与提报规范
* **Bug 表现形式**：少功能、多功能、功能错误、隐性功能错误、难使用。
* **缺陷提交核心要素**：
  * **Bug 标题**：实际结果 vs 预期结果 + 发生位置。
  * **指派人**：对应的开发人员。
  * **严重程度 & 优先级**。
  * **复现步骤 (Steps)**。
  * **附件**：截图、录屏或日志。
* **缺陷跟踪闭环**：提交 Bug -> 开发修复 -> 测试验证 Bug -> 验证通过关闭 Bug / 验证失败重新打开 (Reopen)。

## 五、 App 专项测试
App 除了进行与 Web 端相同的功能测试外，还需覆盖以下专项：
* **安装/卸载/升级**：覆盖不同版本的覆盖安装与数据保留。
* **兼容性测试**：不同机型、系统版本、分辨率兼容。
* **交叉测试 (中断测试)**：【💡 实战重点】App 运行中遇到电话呼入、闹钟、低电量弹窗等，验证 App 是否崩溃或数据丢失。
* **弱网测试**：【💡 实战重点】使用抓包工具（如 Charles/Fiddler）模拟 2G/3G/电梯/地铁网络，测试 App 是否闪退、是否有合理的“加载中”提示及重试机制。
* **Push 消息推送**：验证接收的及时性与跳转准确性。
* **测试环境准备**：模拟器 + APK 打包文件（涵盖内部内测版与线上发布版）。
